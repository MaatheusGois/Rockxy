# Log Intelligence

Rockxy captures application logs alongside network traffic and correlates them by timestamp. When a network request fails, nearby log output can help you line up app behavior with proxy traffic without switching to Console.app.

## Entry Points

| Action | How to Access |
|--------|---------------|
| Logs tab | **Logs** tab in the main content switcher |
| Log capture lifecycle | Starts with the main coordinator when log capture is enabled |
| Correlated request ID | Log entries can be linked to nearby requests by timestamp |

## Capturing Logs

When log capture is enabled, Rockxy streams entries in real time from the configured sources. OSLog entries are pulled from `OSLogStore` (scoped to the current process, polled every 500 ms) and rendered as they arrive.

<Note>
The current Logs tab renders a chronological list with a color-coded severity badge per row. Source, subsystem, level, keyword, and time-range filtering are planned — they are not yet exposed in the UI.
</Note>

## Log Levels

| Level | Description | Color |
|---|---|---|
| Debug | Verbose debugging output, typically disabled in release builds | Gray |
| Info | Informational messages about normal operation | Blue |
| Notice | Notable conditions that are not errors | Cyan |
| Warning | Potential issues that may require attention | Orange |
| Error | Error conditions that caused a failure | Red |
| Fault | Critical system-level failures | Purple |

Levels mirror Apple's OSLog severity ladder. Each entry's badge in the Logs list is colored using the values above.

## Log-Request Correlation

Rockxy currently correlates log entries with network requests using timestamp proximity:

- **Timestamp proximity** — log entries close to a network request are candidates for correlation.

The `correlatedTransactionId` on a log entry links it to the closest request within the correlation window.

<Note>
Correlation accuracy depends on timestamp precision and traffic volume. In busy multi-app sessions, treat the correlation as a helpful proximity signal rather than proof that one process emitted both events.
</Note>

## Log Buffer

Rockxy holds up to **100,000 log entries** in memory for fast access and filtering. When the buffer reaches capacity, older entries are evicted from the live list.

The current Settings UI does not expose a dedicated Logs tab or buffer-size control.

## Log Filtering

Filtering is a planned addition to the Logs surface. Once it lands, you will be able to combine:

- **Level** — show only entries at or above a minimum severity level.
- **Source** — OSLog, stdout/stderr, or custom plugin sources.
- **Keyword** — free-text search across the log message content.
- **Time Range** — restrict to entries within a specific time window.
- **Process** — filter by process name or PID.

Today, the Logs tab shows every captured entry in chronological order; narrow the noise at the source by enabling capture only when you need it.
